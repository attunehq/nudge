# Nudge User Guide

Nudge is a collaborative memory layer for coding agents. It sits in the hook
path for Claude Code and Codex CLI, watches the operations those agents expose,
and reminds them about conventions, local workflow, and previously solved
debugging incidents at the moment the reminder is useful.

The goal is simple: write down the things you keep repeating to agents, then let
Nudge remember them for you.

## What Nudge Does

Nudge has two kinds of memory.

Rules are deterministic conventions. Use them for "never do this", "always use
this command", "this project keeps imports at the top", or "do not fetch remote
docs when local source exists". Rules live in YAML and can match file content,
shell commands, web fetches, or user prompts.

Learned incident notes are repo-local debugging memory. Use them for "we saw
this failure before, here is what happened, here is the fix, and here is how we
verified it". Notes live as plain Markdown under `.nudge/learned/*.md`.

When a match happens, Nudge can:

- Interrupt a risky or convention-breaking tool call before it runs.
- Substitute a simple Bash command with the project-preferred command.
- Continue a prompt with extra context for the agent.
- Surface a learned incident note when the current prompt or tool input looks
  similar to a past debugging session.
- Pass through silently when nothing relevant matches.

## Install

Install the latest release binary.

macOS or Linux:

```bash
curl -sSfL https://raw.githubusercontent.com/attunehq/nudge/main/scripts/install.sh | bash
```

Release binaries support macOS, Linux, and Windows x64. BM25 learned-note
search is always available. Local semantic embeddings are included on Apple
Silicon macOS and x64 GNU Linux; Intel macOS, musl Linux, arm64 GNU Linux, and
Windows GNU builds omit local semantic embeddings.

Windows PowerShell:

```powershell
irm https://raw.githubusercontent.com/attunehq/nudge/main/scripts/install.ps1 | iex
```

From source:

```bash
git clone https://github.com/attunehq/nudge
cd nudge
cargo install --path packages/nudge
```

Confirm the binary is available:

```bash
nudge --help
```

## Set Up A Project

Run setup from the project where you use Claude Code or Codex CLI. Use the
command for the agent you use, or run both if you use both agents.

```bash
nudge claude setup
nudge codex setup
```

Claude setup writes `.claude/settings.local.json` and installs the bundled
`nudge` and `nudge-learnings` skills under `.claude/skills/`.

Codex setup writes `.codex/hooks.json` and installs the bundled `nudge` and
`nudge-learnings` skills under `.agents/skills/`.

Setup does not edit project `CLAUDE.md` or `AGENTS.md` files. Modern agents
learn how to respond to Nudge by reading the bundled skills that setup installs.

If an existing hook file is present, setup writes a non-overwriting backup next
to it before modifying it. Restart open agent sessions after setup so hooks and
skills load. Later rule changes are read by Nudge itself and do not require an
agent restart.

Verify hooks in the agent with `/hooks`.

## Add Rules

Rules are loaded from all of these locations, in addition to one another:

| Location | Scope |
| --- | --- |
| `$CONFIG_DIR/rules.yaml` | User-level rules |
| `.nudge.yaml` | Project root |
| `.nudge.yml` | Project root |
| `.nudge/**/*.yaml` and `.nudge/**/*.yml` | Project rule directory |

Platform config directories:

| Platform | `$CONFIG_DIR` |
| --- | --- |
| Linux | `~/.config/nudge` |
| macOS | `~/Library/Application Support/com.attunehq.nudge` |
| Windows | `%APPDATA%\attunehq\nudge\config` |

Start small. A useful first `.nudge.yaml` is one rule that catches a convention
you currently repeat by hand:

```yaml
version: 1
rules:
  - name: no-unwrap
    description: Require contextual panic messages in Rust
    message: 'Use `.expect("descriptive error message")` instead of `.unwrap()`, then retry.'
    on:
      - hook: PreToolUse
        tool: Write
        file: "**/*.rs"
        content:
          - kind: SyntaxTree
            language: rust
            query: |
              (call_expression
                function: (field_expression
                  field: (field_identifier) @method)
                arguments: (arguments)
                (#eq? @method "unwrap"))
      - hook: PreToolUse
        tool: Edit
        file: "**/*.rs"
        new_content:
          - kind: SyntaxTree
            language: rust
            query: |
              (call_expression
                function: (field_expression
                  field: (field_identifier) @method)
                arguments: (arguments)
                (#eq? @method "unwrap"))
```

The rule-writing reference is generated by the binary:

```bash
nudge claude docs
nudge codex docs
```

Copyable starter rules live in `examples/rules/`.

## Use Nudge

Use your agent normally. Nudge runs through hooks.

If Nudge interrupts, read the message, fix the operation, and retry. Messages
are intentionally direct because they need to cut through while the agent is
focused. A good rule says what is wrong, how to fix it, and that the agent
should retry.

If Nudge substitutes a command, the command proceeds with the rewritten input
and the agent receives context explaining the rewrite.

If Nudge injects context at prompt time, the agent sees the reminder before it
starts the turn.

If no rule or learned note applies, Nudge produces no output.

## What You Can Expect

Nudge is a helper in the loop, not a replacement for review. Expect it to catch
the deterministic things you encode well. Expect it to miss things you have not
encoded, vague rules, and provider surfaces that the current hook APIs do not
expose.

Provider support changes with Claude Code and Codex CLI hook surfaces, but the
current shape is:

| Surface | Claude Code | Codex CLI |
| --- | --- | --- |
| `PreToolUse` `Write` | Yes | Yes, via `apply_patch` add-file parsing |
| `PreToolUse` `Edit` | Yes | Yes, via `apply_patch` update parsing |
| `PreToolUse` `Delete` | Normalized internally | Normalized internally via `apply_patch` delete parsing |
| `PreToolUse` `WebFetch` | Yes | No current WebSearch interception |
| `PreToolUse` `Bash` | Yes | Partial, depending on Codex hook coverage |
| `UserPromptSubmit` | Yes | Yes |
| `PermissionRequest` | Parsed only | Parsed only |

Codex users should write file rules in terms of `Write` and `Edit`; Nudge handles
the `apply_patch` adapter internally. If a Codex patch cannot be inspected,
Nudge allows it and warns the model to report that warning.

## How Agents Use It

Agents interact with Nudge in two ways.

Hook responses are automatic. The agent tries to write, edit, fetch, run a
command, or start a turn. Nudge evaluates rules and learned notes, then returns a
provider-specific response.

The bundled `nudge` skill teaches agents how to respond to hook messages and
write, debug, or validate rules. The bundled `nudge-learnings` skill teaches
agents how to search and record learned incident notes. Setup installs both by
default.

Agents use `nudge` when Nudge blocks, warns, substitutes, or when the user asks
to configure or update rules. Agents use `nudge-learnings` when Nudge surfaces
learned context, when the user asks to search or record learnings, or after
fixing a repo-specific issue that future agents should not rediscover.

Reinstall only the bundled skill files when needed:

```bash
nudge claude skills install
nudge codex skills install
```

## Learned Incident Notes

Record a learning when a debugging session produces knowledge worth reusing.

```bash
nudge learn add --title "Expo Metro resolver cache" --body "What went wrong: Expo could not resolve modules after a dependency update.

Fix: clear the Metro cache and restart the dev server.

Verification: expo start completed and the app loaded."
```

For longer notes:

```bash
cat incident.md | nudge learn add
```

Search notes manually:

```bash
nudge learn list
nudge learn search expo metro cannot resolve module
```

Nudge uses BM25 lexical search by default. Exact error strings, command names,
package names, file paths, and framework names help future agents find the note.

Enable local semantic retrieval for a project:

```bash
nudge learn embeddings enable
nudge learn embeddings status
nudge learn embeddings reindex
```

This stores Markdown notes in the repo and generated model/vector artifacts in
the user-level Nudge cache.

Run `nudge learn embeddings status` before relying on semantic retrieval. If it
prints `Embedding support: unavailable in this binary`, BM25 learned-note search
still works, but `enable` and `reindex` are unavailable. This is expected for
release targets where FastEmbed's ONNX Runtime backend does not publish usable
artifacts, currently Intel macOS, musl Linux, and Windows GNU, or where those
artifacts do not link cleanly in the release cross-toolchain, currently arm64
GNU Linux.

## CI And Programmatic Checks

Use `nudge check` when you want rules enforced outside a live agent session.

```bash
nudge check
nudge check src/
nudge check "**/*.rs"
```

Check mode evaluates file-based block rules for Write/Edit content, including
Regex, SyntaxTree, External matchers, and Markdown code-block targets. It does
not evaluate live hook behavior such as Bash substitutions, WebFetch rules,
prompt reminders, permissions, or learned-note prompt injection.

See [CI and Programmatic Checks](ci.md) for the full check-mode contract.

## Examples

Prompt-time reminder:

```yaml
version: 1
rules:
  - name: dev-server-hint
    message: "Use `npm run dev` for the local server in this repo."
    on:
      - hook: UserPromptSubmit
        prompt:
          - kind: Regex
            pattern: "(?i)start.*(server|dev)|run.*local"
```

Bash substitution:

```yaml
version: 1
rules:
  - name: use-yarn-add
    action: substitute
    on:
      - hook: PreToolUse
        tool: Bash
        command:
          - kind: Regex
            pattern: "^npm install(?: (?P<args>.*))?$"
            replace: "yarn add {{ $args }}"
```

Markdown code-block target:

```yaml
version: 1
rules:
  - name: no-rust-lhs-type-annotations-in-docs
    message: "Use type inference instead of left-hand type annotations in this Rust code block, then retry."
    on:
      - hook: PreToolUse
        tool: Write
        file: "**/*.md"
        target:
          kind: MarkdownCodeBlock
          language: rust
        content:
          - kind: SyntaxTree
            language: rust
            query: "(let_declaration type: (_) @type)"
```

External matcher:

```yaml
version: 1
rules:
  - name: markdownlint
    message: |
      Format this Markdown before continuing.
      Status: {{ $external_status }}
      Pipe the content to `{{ $command }}` to inspect the checker output.
    on:
      - hook: PreToolUse
        tool: Write
        file: "**/*.md"
        content:
          - kind: External
            command: ["markdownlint", "--stdin"]
            timeout_ms: 10000
```

External matchers execute commands from rule YAML, so treat rule files as trusted
local code.

## Troubleshooting

Run `nudge validate` after editing rules.

Run `nudge test` to exercise one rule against sample input.

Run `claude --debug` to inspect Claude Code hook execution.

For Codex, restart sessions after setup, run `/hooks`, and trust the Nudge hook
when prompted. If hooks do not appear, check that the project `.codex/` layer is
trusted and Codex hooks have not been disabled.

Run `nudge learn embeddings status` when learned-note search behaves
unexpectedly. If embeddings are enabled and notes changed substantially, run
`nudge learn embeddings reindex`.
